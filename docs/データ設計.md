# ビジネスマナークイズアプリ データ設計

## 1. 目的

本書は、Next.js 16 で実装するMVPのために必要なデータ構造を定義する。  
問題データと学習履歴を分離し、将来的な問題追加、企業別カスタマイズ、API化に耐える構成を前提とする。

## 2. 設計方針

- 問題コンテンツとユーザー学習データを分離する
- UIから扱いやすいフラット寄りの構造にする
- MVPではローカル保存を前提とする
- 将来のDB移行時もモデル名や責務が大きく変わらないようにする
- 出典、タグ、難易度、適用開始日など拡張項目を持てるようにする

## 3. エンティティ一覧

- Topic
- Question
- Choice
- Source
- QuizSession

- Bookmark

## 4. 型定義

## 4-1. Topic

### 役割
- 問題を学習カテゴリごとにまとめる

### 推奨フィールド
```ts
type Topic = {
  id: string;
  slug: string;
  name: string;
  shortDescription: string;
  description: string;
  sortOrder: number;
  accentColor: string;
  icon?: string;
  isActive: boolean;
};
```

### 例
```ts
{
  id: "T03",
  slug: "keigo",
  name: "敬語・言葉遣い",
  shortDescription: "社会人として自然な敬語を身につける",
  description: "尊敬語、謙譲語、丁寧語、クッション言葉などを学ぶ",
  sortOrder: 3,
  accentColor: "peach",
  isActive: true
}
```

## 4-2. Choice

### 役割
- 4択の選択肢情報を表す

```ts
type Choice = {
  key: "A" | "B" | "C" | "D";
  text: string;
};
```

## 4-3. Question

### 役割
- クイズ1問のコンテンツ本体

### 推奨フィールド
```ts
type QuestionDifficulty = "初級" | "中級" | "上級";

type Question = {
  id: string;
  topicId: string;
  objectiveId: string;
  difficulty: QuestionDifficulty;
  tags: string[];
  question: string;
  situation?: string;
  imageKey?: string;
  choices: Choice[];
  correctAnswer: "A" | "B" | "C" | "D";
  explanation: string;
  incorrectExplanations?: Partial<Record<"A" | "B" | "C" | "D", string>>;
  takeaway?: string;
  sourceIds: string[];
  note?: string;
  version: string;
  effectiveFrom?: string;
  effectiveTo?: string;
  isActive: boolean;
};
```

### 補足
- `objectiveId` は重複管理や学習目標の一意化に使う
- `imageKey` は将来のイラスト差し込みに使う
- `note` には「社内規程による差あり」などを入れる

## 4-4. Source

### 役割
- 問題の根拠となる出典情報を管理する

```ts
type SourceReliability = "A" | "B" | "C";

type Source = {
  id: string;
  title: string;
  category: "公的機関" | "業界団体" | "検定" | "研修会社" | "その他";
  reliability: SourceReliability;
  url: string;
  publishedAt?: string;
  lastCheckedAt?: string;
  note?: string;
};
```

## 4-5. QuizSession

### 役割
- 1回の学習セッションを管理する

```ts
type QuizMode = "normal" | "review" | "unlearned";

type QuizSession = {
  id: string;
  topicIds: string[];
  mode: QuizMode;
  difficulty?: QuestionDifficulty | "おまかせ";
  questionIds: string[];
  currentIndex: number;
  startedAt: string;
  finishedAt?: string;
};
```

## 4-6. Attempt

### 役割
- ユーザーが各問題にどう回答したかを記録する

```ts
type Attempt = {
  id: string;
  sessionId: string;
  questionId: string;
  selectedAnswer: "A" | "B" | "C" | "D";
  correctAnswer: "A" | "B" | "C" | "D";
  isCorrect: boolean;
  responseMs: number;
  answeredAt: string;
};
```

## 4-7. UserProgress

### 役割
- トピック別や全体の学習進捗を保持する

```ts
type TopicProgress = {
  topicId: string;
  answeredCount: number;
  correctCount: number;
  incorrectCount: number;
  accuracyRate: number;
  lastStudiedAt?: string;
};

type UserProgress = {
  totalAnsweredCount: number;
  totalCorrectCount: number;
  totalIncorrectCount: number;
  topicProgressMap: Record<string, TopicProgress>;
  streakDays?: number;
  lastStudiedAt?: string;
};
```

## 4-8. Bookmark

### 役割
- 後で見返したい問題を保存する

```ts
type Bookmark = {
  questionId: string;
  createdAt: string;
  memo?: string;
};
```

## 5. 保存単位

### MVPの保存方法
- `questions.ts` または `questions/*.ts`: 静的問題データ
- `topics.ts`: トピックデータ
- `sources.ts`: 出典データ
- `localStorage`: 学習履歴、進捗、ブックマーク

### localStorageキー例
- `beq.progress`
- `beq.attempts`
- `beq.bookmarks`
- `beq.currentSession`

## 7. MVPで最低限必要な初期データ

### トピック数
- 10トピック程度

### 問題数
- 80〜100問を目安

### 出典数
- トピック横断で20件前後

## 8. 派生データ

UIでは、元データから以下の派生データを算出できるようにする。

- トピックごとの出題数
- トピックごとの正答率
- 未学習問題数
- 要復習問題数
- よく間違えるタグ
- 直近学習日

## 9. 採点・復習ロジックの考え方

### 要復習判定
以下のいずれかに該当したら要復習候補とする。

- 直近回答が不正解
- 同一問題で不正解が複数回ある
- 正解でも回答時間が長い
- ユーザーが明示的にブックマークした

### おすすめ出題
- 未学習問題を優先
- 次に不正解問題を優先
- 同じ問題が連続しすぎないようにする
- トピック偏りを避ける

## 12. 実装上の注意

- 初期段階では、型安全性を優先して TypeScript のデータ定義で管理する
- 問題データはUIコンポーネント内に直書きしない
- 出典情報を独立管理し、同じURLを問題ごとに重複記述しない
- 将来的にCMSへ移行する場合も、現モデルを近い形で維持する
