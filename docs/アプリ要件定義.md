# ビジネスマナークイズアプリ 要件定義

## 1. 概要

### 1-1. 目的
新卒女性社員向けに、日本のビジネスマナーをスマートフォンでも学びやすいクイズ形式で習得できる研修アプリを提供する。  
単なる正誤判定ではなく、「なぜそれが適切なのか」を理解できる解説を通じて、現場で再現できる行動変容につなげる。

### 1-2. 前提
- 対象: 新卒女性社員研修用
- UI方針: 女性向けで可愛い印象。ただし業務研修アプリとして幼すぎず、清潔感と信頼感を両立する
- 対応端末: スマートフォン優先、タブレット・PCでも利用可能なレスポンシブ対応
- 実装基盤: Next.js 16
- 利用言語: 日本語

### 1-3. 成功イメージ
- 新卒社員が短時間で基礎マナーを反復学習できる
- 研修担当者が「基礎知識の定着度」を把握できる
- 実務で迷いやすい敬語、電話、メール、名刺交換、席次などを重点的に訓練できる

## 2. 背景

既存の調査資料では、日本のビジネスマナーは以下のように整理できる。

- 基礎姿勢: 身だしなみ、表情、姿勢、時間厳守
- 対人所作: 挨拶、お辞儀、来客対応、訪問対応
- 言語配慮: 敬語、クッション言葉、相手視点の表現
- 実務プロトコル: 電話、メール、名刺交換、会議、報連相
- コンプライアンス: ハラスメント防止、個人情報保護

上記のうち、新卒研修で即戦力化に直結する領域をMVPで優先し、接待・慶弔などは拡張範囲として段階導入する。

## 3. ターゲットユーザー

### 3-1. 主対象
- 企業に入社したばかりの新卒女性社員
- 社会人経験が浅く、ビジネスマナーに不安があるユーザー
- 通勤中や研修の合間にスマホで短時間学習したいユーザー

### 3-2. 関係者
- 研修担当者
- 人事担当者
- 配属先の上司やOJT担当者

### 3-3. ペルソナ
- 22歳、総合職新卒
- 敬語、電話応対、名刺交換に特に不安がある
- 「厳しすぎる教材」より、やさしい見た目と分かりやすい説明の方が継続しやすい

## 4. プロダクトゴール

### 4-1. 学習ゴール
- 新卒社員が基本的なビジネスマナーを判断できる
- 状況に応じて適切な行動や表現を選べる
- 間違えた理由を理解し、次回正しく振る舞える

### 4-2. 業務ゴール
- 研修初期に起きやすいマナー事故を減らす
- 現場配属前に最低限の共通理解を揃える
- 研修コストの一部をデジタル教材で代替する

## 5. 対象範囲

### 5-1. MVPで含める範囲
- 基本姿勢・身だしなみ・時間厳守
- 挨拶・お辞儀・第一印象
- 敬語・言葉遣い
- 電話応対
- メール・文書作法
- 名刺交換・来客対応・訪問対応
- 会議・席次・打ち合わせの基本
- 報連相・社内コミュニケーション
- ハラスメント防止の基礎
- 個人情報・情報漏えい防止の基礎

### 5-2. Phase 2以降で検討する範囲
- 接待・贈答
- 慶弔マナー
- テレワーク特有のマナー
- 管理者向けダッシュボード
- 企業ごとの独自ルール差し替え

## 6. 機能要件

### 6-1. 学習機能
- トピック別にクイズを出題できること
- 4択問題を基本形式として提供すること
- 問題回答後に即時で正誤判定を表示すること
- 各問題に解説、理由、注意点を表示すること
- 出典または根拠カテゴリを表示できること
- 初級 / 中級 / 上級の難易度を持てること
- 不正解問題のみを復習できること
- ブックマークした問題を後で見返せること

### 6-2. 学習導線
- 初回利用時に「おすすめ学習順」を提示できること
- 1回5問、10問など短時間学習モードを選べること
- 学習完了後にスコア、正答率、弱点カテゴリを表示できること
- 継続学習しやすいよう、次に学ぶべきトピックを提案できること

### 6-3. 記録機能
- 回答履歴を保存できること
- トピックごとの正答率を保持できること
- 最終学習日時を保存できること
- 復習対象問題を抽出できること

### 6-4. 管理・運用機能
- 問題データを構造化して管理できること
- 問題にトピック、難易度、タグ、出典IDを紐づけられること
- 将来、問題追加や差し替えがしやすいこと

## 7. コンテンツ要件

### 7-1. 問題構成
- 問題はトピック別に整理する
- 問題文は新卒社員が想像しやすい実務シーンで作成する
- 正解だけでなく、不正解がなぜ不適切かも説明する
- 暗記だけでなく、状況判断を問う設問を含める

### 7-2. MVPトピック案
- T01: 基本姿勢・時間・身だしなみ
- T02: 挨拶・お辞儀・第一印象
- T03: 敬語・言葉遣い
- T04: 電話応対
- T05: メール・文書
- T06: 名刺交換・訪問・来客対応・席次
- T07: 会議・打ち合わせ
- T08: 報連相・社内コミュニケーション
- T09: ハラスメント防止
- T10: 個人情報・情報漏えい防止

### 7-3. 問題数の目安
- MVP初期は各トピック10問前後、合計80〜100問を目安とする
- 将来的には既存資料の120問問題バンクを取り込み拡張できる構成とする

### 7-4. 解説方針
- ルールの暗記ではなく、相手への配慮や社会人としての意味を伝える
- 法令や指針に関わる領域は、一般マナーと区別して明示する
- 企業差・業界差があり得る内容は「社内ルール確認」の注記を入れる

## 8. UI/UX要件

### 8-1. デザインコンセプト
- かわいい、やわらかい、親しみやすい
- ただし研修利用に耐えるよう、上品さと読みやすさを優先する
- 「女性向け」を、配色・余白・イラスト・丸み・柔らかいトーンで表現する

### 8-2. ビジュアル方針
- メインカラーは淡いピンク、アイボリー、コーラル、ミントなどのやさしい配色
- 補助色にネイビーやグレージュを使い、甘すぎない印象に整える
- カードUI、丸みのあるボタン、やわらかなシャドウを採用する
- イラストやアイコンは線が細めで清潔感のあるタッチを採用する

### 8-3. 画面体験
- 1画面1目的を徹底し、スマホで迷いにくい導線にする
- 文字サイズ、行間、余白を十分に確保する
- 回答ボタンは親指で押しやすいサイズにする
- 正解/不正解は色だけでなく文言やアイコンでも区別する
- 学習の達成感を感じられる進捗表示を入れる

### 8-4. イメージ素材
- 必要に応じて、シーンイラストやマナー解説用画像を生成しながらUIに組み込める設計にする
- 画像はスマホ表示でも情報が潰れない構図を優先する
- 画像未使用でも成立するUI構造にする

## 9. 画面要件

### 9-1. MVP画面一覧
- トップページ
- 学習トピック一覧
- クイズ出題画面
- 回答結果・解説画面
- 学習結果サマリー画面
- 復習一覧画面
- マイページまたは学習状況画面

### 9-2. 各画面の要点
- トップページ: 今日のおすすめ学習、進捗、主要トピックへの導線を表示
- トピック一覧: 各カテゴリの概要、問題数、達成率を表示
- クイズ画面: 問題文、選択肢、進捗、戻る導線を明確に表示
- 解説画面: 正誤、理由、実務での注意点、復習登録導線を表示
- サマリー画面: 正答率、苦手分野、次のおすすめを表示

## 10. レスポンシブ要件

### 10-1. 対応方針
- スマホファーストで設計する
- 最低対応幅は 360px を想定する
- 主要確認幅は 360px / 390px / 430px / 768px / 1024px とする

### 10-2. スマホ利用前提の要件
- 片手操作でも主要アクションが完結すること
- 重要情報はファーストビュー内に優先表示すること
- 横スクロールが発生しないこと
- 画像や表はモバイルで読みやすい代替レイアウトを用意すること

## 11. 非機能要件

### 11-1. パフォーマンス
- 初回表示はモバイル回線でも快適に感じられること
- 画像は最適化し、不要な大容量アセットを避けること
- クイズ操作時に体感的な遅延を出さないこと

### 11-2. アクセシビリティ
- 色だけに依存しない情報設計を行うこと
- ボタンやフォームに十分なタップ領域を確保すること
- 画面読み上げでも主要操作ができる構造を目指すこと

### 11-3. 保守性
- 問題データはUIと分離して管理すること
- コンテンツ追加時にコード改修が最小で済むこと
- 将来のトピック追加や企業別カスタマイズに耐えられる構成にすること

## 12. 技術要件

### 12-1. フロントエンド
- Next.js 16 を採用する
- App Router ベースで構成する
- TypeScript を前提とする
- UIコンポーネントは再利用可能な設計にする

### 12-2. データ設計
- 問題データは JSON または TypeScript 管理の構造化データで保持する
- 最低限、以下の情報を持つ
  - questionId
  - topicId
  - difficulty
  - question
  - choices
  - correctAnswer
  - explanation
  - tags
  - sourceIds

### 12-3. 状態管理
- 学習進捗、履歴、ブックマーク、復習対象を管理できること
- MVPではローカル保存を基本とし、将来的にAPIやDBへ移行しやすい構造にする

## 13. 推奨アーキテクチャ

### 13-1. 初期構成
- `app/`: 画面ルーティング
- `components/`: 共通UI
- `features/quiz/`: クイズ出題と判定ロジック
- `features/progress/`: 学習履歴と進捗表示
- `data/questions/`: 問題データ
- `lib/`: データ整形、採点、フィルタ処理

### 13-2. データモデル例
- Topic
- Question
- Choice
- Attempt
- Bookmark
- Source

## 14. MVP受け入れ条件

- スマホで主要画面が崩れずに利用できる
- 10カテゴリ前後のトピックが表示できる
- 4択クイズが出題できる
- 正誤判定と解説表示ができる
- 回答履歴と復習対象を保持できる
- 可愛い印象のUIでありながら、研修用途として読みやすい
- Next.js 16 ベースで構成されている

## 15. リスクと留意点

- 「可愛いUI」を優先しすぎると、研修アプリとしての信頼感が落ちる可能性がある
- マナーには一般論と社内独自ルールが混在するため、断定表現には注意が必要
- ハラスメントや個人情報領域は法令・指針改定に追随できる設計が必要
- 画像依存のUIにすると制作コストが上がるため、テキスト主体でも成立する設計が望ましい

## 16. 現時点の前提・仮置き事項

- 対象ユーザーは日本国内企業の新卒女性社員を想定する
- まずは社内研修向けの単一テナント想定とする
- 認証機能や管理者向け配信機能はMVP外とする
- 初期データは既存PDF資料の問題バンクをベースに整備する
- デザイン制作時に必要であれば、アプリ内イラストやイメージ画像を生成して組み込む

## 17. 次に決めるべきこと

- ログイン有無
- 研修担当者向け管理画面の要否
- 問題データをローカル管理にするかCMS連携前提にするか
- ブランドトーンを「可愛い寄り」か「可愛い + きれい寄り」かどちらにするか
- MVPで本当に含めるトピック数と問題数
